---
title: "ADR 0003: In-process circuit breaker"
description: "Why the provider circuit breaker keeps state in-process for now, with Redis deferred."
---

**Status:** Accepted (Redis-backed state deferred to a later performance stage)

## Context

ProBot calls third-party LLM providers on the hot path. When a provider degrades, hammering it with requests wastes the user's quota, slows every request behind a timeout, and can deepen the provider's outage. A **circuit breaker** trips after repeated failures and fails fast instead. The open question was *where the breaker's state lives* - in-process memory or a shared store like Redis.

## Decision

Keep the breaker **in-process**, keyed per provider, with a standard `closed → open → half-open` state machine:

- **closed:** calls flow; consecutive failures are counted, a success resets the count.
- **open:** after N consecutive failures, every call rejects immediately with `circuit_open` until a reset timeout elapses.
- **half-open:** the first call after the cooldown is allowed through as a probe; success closes the circuit, failure re-opens it. Concurrent calls beyond the probe count are rejected.

## Alternatives considered

- **Redis-backed shared state** - gives correct behaviour across multiple instances (all instances see the same open circuit). Rejected *for now*: it adds an external dependency and operating surface, and the cost only pays off at multi-instance scale. Explicitly deferred to the performance/scale stage, where the rate limiter moves to shared state too.
- **No breaker, rely on timeouts** - simpler, but offers no fast-fail and no back-off, so a degraded provider drags down every request. Rejected.

## Consequences

- Zero added infrastructure: the breaker is pure in-memory state.
- On serverless/Vercel, cold starts naturally bound the memory and reset the breaker. For an upstream that recovers in minutes (not hours), a per-process breaker is acceptable.
- **Trade-off:** in a multi-instance deployment each instance maintains its own breaker, so they trip independently rather than in unison. Accepted until shared-state rate limiting lands, at which point the breaker can adopt the same Redis store.
